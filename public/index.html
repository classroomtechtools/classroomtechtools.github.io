Proposal

Standard Libraries for Google AppsScripts

Observations

Regarding the concept of citizen developers (based on the community forums)

  • They struggle with “simple” patterns, such as iterating over data structures.
  • They struggle with commonly used design problems, such as how to use a cache
  • There’s some more advanced coders who would like to make it easier for these citizen developers to use their code

Identifiers

The platform’s runtime already provides developers with identifiers with very cool code completion.

Some are built-in, some need to be specifically enabled:

function myFunction () {
	const ss = SpreadsheetApp.getActiveSpreadsheet();
	const cache = CacheService.getUserCache();

	const response = Sheets.Spreadsheets.Values.get('<id>', 'Sheet1!A1:D5');
}
```

Libraries

Libraries are a great way to share code and problem-solving, but. . .

  • ES modules are not supported
  • Instead they are imported with ID code
  • No search, or community-guided recommended ones

Other ecosystems

Node, Python, Ruby all have mechanisms for end user developers to build upon more advanced coding sleuths to provide them with libraries with abstractions that solve problems.

Solutions

Create a way for even Citizen Developers to build upon the expertise of some more advanced coders, by allowing for “standard libraries”

Standard Libraries

A group of libraries that have been vetted by the community as providing dependable, reliable, and easy-to-understand abstractions that citizen developers can use.

How do we access them?

Possible Implementations

1) Use existing Library system

Make it so that there is a list of some libraries, and the end user adds them “the regular way”

const Range = Identifier();  // chosen by user

Pros/Cons

Pros:

  • Relies on the build-in way to add a library

Cons:

  • JSDoc does not work well with autocomplete

Copy and paste a file

We could build a github repo where they can choose the libraries, and then copy and paste it into their project.

Pros/Cons

Pros:

  • End users get the source code and can fiddle with it easily
  • Great autocomplete support

Cons:

  • These libraries would be populating the global namespace
  • The order of parsing for files is important if they use in global context

Reserved Namespace

Define a reserved word as a namespace, like Import, we could build libraries on that as a namespace.

// "import" it
const {Range} = Import;

// define
const range = Range.fromA1Notation('Sheet1!A1:C', {dataRange: true});

// use (here, as an iterator)
for (const row of range) {
	for (const cell of row) {
        // object `cell` has methods/properties
		cell.a1Notation;
		cell.value;
	}
}

Pros/Cons

Pros:

  • Introduces the idea of importing stuff
  • Great autocomplete support
  • Does not pollute global namespace

Cons:

  • Libraries need to be pre-coded so that the Import object is mapped with libraries
  • Still potential for name conflicts, which can be solved by convention
